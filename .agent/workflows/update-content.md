## Update Content Workflow

### 1. Make your content changes

Edit files in `src/` depending on what you need:

- `src/_data/experience.json` — work experience entries
- `src/_data/projects.json` — project entries
- `src/_data/certifications.json` — certification entries
- `src/uses.vto` — the /uses page
- `src/index.vto` — homepage
- `src/styles/main.css` — styles (LightningCSS, supports nesting and variables)
- `src/blog/NNN-kebab-title.md` — blog posts (see `docs/ADDING_BLOG.md`)

### 2. Preview locally (optional)

```bash
deno task dev
```

This starts Lume with live reload. The site is served from `_site/` (generated, gitignored).

### 3. Lint and type-check before committing

```bash
deno task lint && deno task check
```

These run automatically as part of `deno task build` but it's good to catch issues early.

### 4. Build for production (optional local verification)

```bash
deno task build
```

This runs `deno audit` then a production Lume build (minifyHTML enabled) into `_site/`.
The service worker (`sw.js`) is generated by `scripts/build-sw.js` and reads the current
git SHA for automatic cache-busting — run `deno task build:sw` separately if needed.

### 5. Deploy to Vercel

Vercel deploys automatically on push to `main`. Just commit and push:

```bash
git add -A
git commit -m "feat: describe your change"
git push
```

Or use the `/deploy` OpenCode command:

```
/deploy feat: describe your change
```

### Notes

- **Never commit** `resume.json`, `main.typ`, or anything in `_cache/` — they are gitignored
- To update the resume PDF, use `deno task resume` then `git add src/resume.pdf`
- Build output goes to `_site/` (gitignored) — Vercel builds from source on its own infrastructure
- The service worker uses a git SHA-based cache version — no manual cache busting needed
